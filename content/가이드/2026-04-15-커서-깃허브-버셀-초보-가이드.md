## 1. 이 프로젝트가 하는 일 (한 줄)

**내 PC 폴더에 있는 마크다운 글**을 읽어서 **웹사이트처럼 보여 주는 블로그**를 만든 뒤, **GitHub에 올리고**, **Vercel이 그 코드를 받아 인터넷에 공개**할 수 있게 만든 구조입니다.

---

## 2. Cursor 에이전트는 뭘 했나?

**에이전트**는 Cursor 안에서 **당신의 요청을 이해하고**, **파일을 읽고·쓰고·터미널 명령을 실행**하는 AI입니다. 사람이 할 일을 대신하되, **도구**를 써서 합니다.

### 2.1 어떤 도구를 썼는지 (이 블로그 작업 기준)

| 도구 종류 | 하는 일 | 예시 |
|-----------|---------|------|
| **파일 읽기** | 코드·설계 문서 확인 | `Design.md`를 읽고 색·글꼴 규칙 반영 |
| **파일 검색** | 프로젝트 안에서 글자 찾기 | 특정 문구가 있는 파일 찾기 |
| **파일 작성/수정** | 코드·설정·글 추가 | `src/app/page.tsx`, `content/...md` 생성 |
| **터미널 실행** | 빌드, npm 설치, git | `npm install`, `npm run build`, `git push` |

즉, 에이전트는 **에디터와 터미널을 대신 조작하는 조수**에 가깝습니다.

### 2.2 단계별로 무슨 일이 있었는지 (쉬운 예)

**① 블로그 만들기 요청**  
에이전트가 Next.js 프로젝트 구조를 만들고, `content/주제/날짜-제목.md` 규칙으로 글을 읽는 코드를 씁니다.  
→ **비유**: 레고 설명서 보고 “블로그 세트”를 조립.

**② 디자인 적용**  
`Design.md`를 읽고 검은 배경·네온 색 같은 스타일을 CSS/Tailwind에 반영합니다.  
→ **비유**: 인테리어 도면을 보고 벽지 색 맞추기.

**③ 내비게이션 고정**  
헤더에 `sticky` 같은 CSS를 넣어 스크롤해도 상단에 붙게 했습니다.  
→ **비유**: 메모를 “항상 보이는 위치”에 고정.

**④ 샘플 글 작성**  
`content/하네스/...md` 같은 파일을 새로 씁니다.  
→ **비유**: 공책에 예시 글 한 편 적어 두기.

**⑤ 이름 바꾸기 (“포피스의 공방”)**  
화면 제목·브라우저 탭·헤더 로고 문구를 찾아서 바꿉니다.  
→ **비유**: 간판 글자만 교체.

**⑥ GitHub에 Push**  
`git init` → 파일 모으기(`add`) → 기록(`commit`) → 원격 연결(`remote`) → 업로드(`push`).  
→ **비유**: 클라우드 폴더(GitHub)에 백업하되, **언제 무엇을 바꿨는지** 기록이 남음.

---

## 3. GitHub는 무슨 역할인가?

**GitHub = 코드와 글의 “중앙 저장소(클라우드 + 타임머신)”**입니다.

### 3.1 쉬운 비유

- **내 PC 폴더**: 지금 작업하는 책상.  
- **GitHub 저장소**: 내용을 **복사해 두는 곳**인데, **“언제 무엇을 바꿨는지” 기록(커밋)**이 남습니다.

### 3.2 왜 쓰나?

1. **백업**: PC가 고장 나도 GitHub에서 다시 받을 수 있음.  
2. **협업**: 나중에 같이 고칠 때 이력을 공유.  
3. **배포의 출발점**: Vercel이 **GitHub에 올라온 최신 코드**를 자동으로 가져가서 빌드함.

### 3.3 `push`는?

**로컬(내 PC)** 에 만든 커밋을 **원격(GitHub)** 의 `main` 브랜치로 **올리는 것**입니다.

---

## 4. Vercel은 무슨 역할인가?

**Vercel = “GitHub에 있는 Next.js 프로젝트를 받아서, 빌드하고, URL로 세상에 공개하는 호스팅”**입니다.

### 4.1 쉬운 비유

GitHub에 **레시피와 재료(소스 코드)** 를 올려 두면,  
Vercel이 **주방에서 요리(빌드)** 해서 **손님에게 나가는 접시(웹사이트)** 로 만듭니다.  
손님은 **`https://….vercel.app`** 같은 주소로 접속합니다.

### 4.2 연결했을 때 흐름

1. GitHub에 `push`  
2. Vercel이 새 커밋을 감지  
3. 서버에서 `npm install` → `npm run build` 실행  
4. 성공하면 새 버전이 공개 URL에 반영

### 4.3 GitHub vs Vercel 한 줄 정리

| | GitHub | Vercel |
|---|--------|--------|
| **주 역할** | 코드·이력 보관 | 그 코드로 **실제 웹사이트 실행·공개** |
| **비유** | 창고 + 변경 기록부 | 공연장 + 무대 세팅 |

---

## 5. 전체 파이프라인 (암기용)

```text
마크다운 글(content/) + Next.js 코드(src/)
  → 로컬에서 npm run dev 로 확인
  → git commit 후 GitHub에 push
  → Vercel이 GitHub를 보고 빌드·배포
  → 브라우저로 접속
```

---

## 6. 다음에 스스로보면 좋은 것

1. **글 추가**: `content/새주제/YYYY-MM-DD-제목.md` 만들고 로컬에서 새로고침.  
2. **배포 확인**: GitHub에 push한 뒤 Vercel 대시보드에서 빌드 로그 읽기.  
3. **에러 날 때**: Vercel 빌드 로그의 **첫 빨간 에러 한 줄**을 검색하거나 에이전트에게 붙여 넣기.
